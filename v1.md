## Core architecture

```
[ Raw data input ]
       ↓
[ Data loader & normalizer ]
       ↓
[ Baseline learner ]  ←  learns what "normal" looks like
       ↓
[ Anomaly detector ]  ←  flags what deviates
       ↓
[ Explainer ]         ←  says why in plain language
       ↓
[ Output: flagged list + confidence + reason + chart ]
```

Each of these is a separate module. They can be swapped, upgraded, or replaced independently. The detector layer can grow from statistical → ML → hybrid without touching the loader or the explainer.

## What to build, in exact order

### Phase 1 — Data loader (`loader.py`)

This is the least glamorous part and the one most projects underinvest in. Do it properly.

**What it does:**
- Reads CSV and JSON
- Handles missing values (imputation or row dropping, configurable)
- Normalizes columns to comparable scales
- Detects whether the data is time-series (has a timestamp column) or purely tabular
- Returns a clean `pandas` DataFrame with metadata attached

**Why first:** Every other module depends on clean input. If your data loader is fragile, your entire engine is fragile.

**Test it with:**
- A CSV with missing values
- A JSON with nested fields
- A dataset with mixed types (strings, floats, dates in the same file)

**You're done when:** You can load any reasonably structured CSV and get back a clean, normalized DataFrame in under 3 lines of calling code.

---

### Phase 2 — Baseline learner (`baseline.py`)

The engine needs to know what "normal" looks like before it can flag anything. This module learns it.

**What it does:**
- For time-series data: computes rolling mean and rolling standard deviation over a configurable window (e.g., 30 data points)
- For tabular data: fits a distribution per column (mean, std, percentile boundaries)
- Stores the baseline as a reusable profile (can be saved to JSON so you don't recompute on every run)

**Why this matters:** Anomaly detection without a baseline is just thresholding. The baseline is what makes your system adaptive — a spike in December stock volume isn't anomalous if December is always high. Context is everything.

**Test it with:**
- Yahoo Finance data (stock prices) — rolling 30-day baseline
- Any dataset where you manually know a "typical" range

**You're done when:** You can print the baseline profile for any column and it matches your intuition about the data.

---

### Phase 3 — Anomaly detector (`detector.py`)

The core. Build this in three sub-layers, each additive:

**Layer 1 — Statistical methods**
- Z-score: flag any point more than N standard deviations from the mean (start with N=3)
- IQR method: flag points outside [Q1 − 1.5×IQR, Q3 + 1.5×IQR]
- These are fast, transparent, and work immediately with no training

**Layer 2 — ML-based (Isolation Forest)**
- Uses `scikit-learn`'s `IsolationForest`
- Unsupervised — no labels needed
- Works on high-dimensional tabular data natively
- Returns an anomaly score per row (negative = more anomalous)
- Start with `contamination=0.01` (assumes ~1% of data is anomalous) and make it configurable

**Layer 3 — Rule-based**
- User-defined thresholds: "flag any transaction above 10,000"
- Domain-specific guard rails that sit on top of statistical/ML detection
- Implemented as a simple config dict or YAML file

**Run all three and combine results.** A point flagged by two or more methods gets higher confidence. This is your first version of ensemble detection.

**Three anomaly types to detect:**
1. **Point anomaly** — a single data point that's wildly out of range
2. **Contextual anomaly** — a value that's normal globally but abnormal in context (e.g., high weekend traffic on a B2B platform)
3. **Pattern shift** — the distribution itself has moved (rolling mean drifts significantly over time)

**You're done when:** Running the detector on the credit card fraud dataset returns fraud cases at a meaningfully higher rate than random.

---

### Phase 4 — Explainer (`explainer.py`)

This is what separates your engine from a script someone wrote in 20 minutes.

**What it does:**
- For every flagged row, generates a plain-language sentence explaining the flag
- Examples:
  - `"Row 847: transaction_amount is 4.2σ above the 30-day rolling mean (mean: 312.4, flagged value: 9,847.0)."`
  - `"Row 1204: Isolation Forest anomaly score −0.31, among the most anomalous 0.8% of records."`
  - `"Row 23: value exceeds user-defined threshold (rule: amount > 10000)."`
- Attaches a confidence score: `0.0 – 1.0` based on how many methods flagged it and how far it deviates

**Why this is non-negotiable:** If the output is just a list of row numbers, no one can use it — including you. The explanation is what makes the system honest and accountable. It also directly aligns with your ethical goals: you're not producing opaque outputs, you're producing reasoning.

**You're done when:** Every flagged row has a reason attached that a non-technical person could understand in 10 seconds.

---

### Phase 5 — Output + visualization (`main.py` + `visualizer.py`)

**CLI interface:**
```bash
python main.py --input data.csv --method all --output results.json
```

Flags: `--method` (statistical / isolation_forest / rules / all), `--window` (rolling window size), `--threshold` (z-score cutoff), `--contamination` (for Isolation Forest)

**Output format (JSON):**
```json
[
  {
    "row": 847,
    "timestamp": "2024-03-15",
    "column": "transaction_amount",
    "value": 9847.0,
    "anomaly_score": 0.91,
    "methods_flagged": ["statistical", "isolation_forest"],
    "explanation": "transaction_amount is 4.2σ above 30-day mean"
  }
]
```

**Visualization:**
- Time-series plot with anomalies marked as red dots (matplotlib)
- Export as PNG or interactive HTML (use `plotly` for HTML if you want interactivity)
- Keep it simple — the plot should be readable in 5 seconds

**You're done when:** You can run one command and get a JSON file + a chart, both in under 10 seconds, on a dataset you've never seen before.

## What to test (and how to know it's working)

### Correctness tests
- Run on the credit card fraud dataset. Does your engine flag actual fraud cases at a higher rate than random? (Even 2× better than random is meaningful at v1.)
- Inject a synthetic spike into a clean dataset. Does it get caught?
- Run on a clean dataset with no anomalies. Does it stay quiet? (False positive rate matters.)

### Robustness tests
- Feed it a dataset with 30% missing values. Does it handle it without crashing?
- Feed it a single-column dataset. Does it still work?
- Feed it a 100,000-row dataset. Does it complete in under 60 seconds?

### Explainability tests
- Take any flagged row. Read the explanation aloud. Does it make immediate sense?
- Show the output to someone who didn't build it. Can they understand which rows are suspicious and why?

### Domain tests
- Run the exact same `main.py` command on a finance dataset, then on a network log, then on stellar light curve data. Does it produce sensible output each time without any code changes?

**If the answer to all of the above is yes — v1 is done.**

## What you should want in the longer term (what v1 seeds)

These are not v1. Do not build them now. But know they're waiting.

### Near-term extensions (v2 territory)
- **REST API wrapper** — expose the engine as a local API so other tools can call it
- **Streaming data support** — detect anomalies in real-time as data arrives (Kafka, websockets)
- **Model persistence** — save the Isolation Forest model so you don't retrain on every run
- **Multi-column contextual detection** — flag anomalies that are only visible when you look at two columns together (e.g., high amount + unusual merchant + foreign country)

### Medium-term (v3 / separate project)
- **Domain-specific modules** — a `SpacetechDetector` that understands light curves, a `FinanceDetector` that understands transaction velocity
- **Visualization dashboard** — a local web UI (Flask/FastAPI + a simple frontend) to explore anomalies interactively
- **Feedback loop** — let a user mark a false positive, have the engine learn from it

### Long-term (the vision you described)
- **Media provenance system** — anomaly detection on metadata signatures to flag AI-generated content. The engine you build in v1 is the same engine that detects statistical irregularities in image metadata, audio spectrograms, and video frame distributions.
- **Trust and reputation layer** — flag anomalies in behaviour patterns (not just data points). An account that suddenly changes transaction patterns, a node that starts routing unusually — same engine, new domain.
- **Fraud detection at scale** — the Isolation Forest you train in v1 is the same family of models used in production fraud systems. You're not building a toy; you're learning the real approach.
- **Spacetech anomaly classification** — flag unusual stellar objects, detect orbital drift in satellite telemetry, identify anomalous light curves that might indicate exoplanets. The v1 engine handles this already; the long-term version adds domain-specific feature engineering.

### The ethical throughline
Every extension of this engine is in service of surfacing truth, not manufacturing it. You're not building something that exploits attention, creates dependency, or hides its reasoning. The explainability requirement baked into v1 means that every future version is inherently accountable. That's not just a nice value — it's a design constraint that makes the system better.
